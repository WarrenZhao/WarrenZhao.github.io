第一个项目HelloWorld

  1. 加入hibernate核心的jar包
  2. 引入数据库的驱动（jar包）
  3. 在数据库中新建相应的表
  4. 在项目中新建类，和数据表中列一一对应
  5. 参考原始文档，新建hibernate配置文件，默认放在src包中，最好不要改hibernate.cfg.xml的名字。包括连接设置、方言、二级缓存、生成SQL设置以及一个mapping就能使用了。
  6. 建立相应类Student
  7. 建立Model，和实体类放在一起，Student.hbm.xml映射文件，进行设置。设置class、id（主键）和其他的属性。
  8. 在hibernate.cfg.xml中设置mapping。
  9. 编写业务代码，读取配置文件使用hibernate的Configuration类。读取配置文件，创建session工厂，通过工厂打开session，这个session是hibernate中的session。然后进行session操作。可以使用HibernateUtil辅助类（单例，写不写没关系）

总结：关键是建立关联。

建立Annotation版本的HelloWorld

  1. 加入hibernate-annotation的相关jar包（3个）
  2. 在实体类中类定义上面加入@Entity的注释（是javax.persistence.Entity的），适用于类名和数据库表名（一般不区分大小写）一致的
  3. 然后给主键加注解。
  4. 在hibernate.cfg.xml中加入相关的mapping标签。
  5. 业务代码中，和上面唯一不一致的就是Configuration要New AnnotationConfiguration。

注意：项目出了莫名其妙的不一致inconsistent的问题重命名一下试试。

搭建Junit日志环境

  1. 添加类库。
  2. 测试代码放在src同级的test中，专门放test的源码，测试哪个类，就在test里面添加一个和src一样的类包。

hibernate常用配置

  1. 表名和类名不一致的时候，使用@Table注解。
  2. 字段名和属性名不一致的时候使用@Column注解，xml用column属性。
  3. 不需要加入到数据库字段中的，在get、set方法上面加@Transient注解即可，在xml中不写即可。
  4. 时间和日期的配置，java类中一般使用java.util.date，在持久化的时候，映射使用datetime，使用@Temporal注解，但是用的并不多。
  5. 映射枚举类型@Enumeration注解即可，在枚举使用的时候加入，可以选择type和ordinal两种配置。

联合主键

在数据库设计的时候，最好不要使用联合主键，使用的时候很有可能是解决遗留问题。

在程序代码中，将主键设计成一个单独的类。而在数据库中，表结构并没有发生变化。

类初始化好之后，更新配置文件。

主键类要实现序列化，即实现java.io.Serializable接口。这样做的原因主要是为了：1.集群中传输对象的需要；2.单机服务器中，内存和虚拟内存交换数据的需要。同时，要重写equals方法和hashCode两个方法。

两种方式的配置实现（xml方式和Annotation的方式）。

核心开发接口

SessionFactory的方法中，openSession和getCurrentSession的区别：

  1. openSession永远打开新的session；getCurrentSession在当前上下文中找，如果session没有提交打开已有的session，一旦commit之后打开新的。
  2. openSession需要手动close使用的session，如果是getCurrentSession每次commit之后自动close无需手动关闭。
  3. 使用getCurrentSession必须要设定session的上下文。

应用场景举例：用户管理模块。

Client端进行访问，UserManager负责user的增删改，这个时候，就要考虑事务的边界问题。

商城订单举例

一个下单的操作可能会涉及两个表：商品表和财务表。会用到事务的一致性。

Session接口

save方法

Hibernate里面对象的状态：

  1. Transient：刚new出来没有id，内存中的一个对象，缓存中没有
  2. Persistent：调用了save，已经有id了，缓存中有，数据库中也有
  3. Detached：session使用closed方法后，缓存没有，数据库有。

delete方法

不论对象状态如何，都可以，要以最后的实验的结果。

load方法

从数据库中取得记录，转换成对象，放到内存中。产生代理对象，使用的时候才发出sql语句。

get方法

直接从数据库中加载，发出SQL语句。

update方法

detached对象可以update为persistent对象。
但是transient对象不能使用update，因为没有id，但是如果手动设置了id，那么久不会报错了，报错的原因就是数据库中没有id。

有用的部分：只update部分更新，因为默认是更新全部字段。

persistent对象只要修改，commit就会update。

其实忘了就做实验试试不就行啦。

如何进行部分更新？

  1. 使用注解column，将参数updatable设置为false即可；
  2. xml设置update属性为false；
  3. 直接使用hql语句（建议）。

saveOrUpdate

执行save或者update看实际需求。

clear

每次从数据库中拿了对象，会进行缓存，使用clear方法，可以清除缓存。

注意：clear不涉及数据库，直接清除缓存。

flush

每次flush的时候，强制让缓存和数据库进行同步。

schemeExport

默认的字段的长度，约束，可以进行设置。

关系映射

对象之间的关系：

  1. 一对一
  2. 一对多
  3. 多对多

加上方向又会有：

  1. 单向
  2. 多向

集合映射、集成关系（很少用）、组件映射

对象和对象之间如何关联？在对象里面添加另外一个对象的引用，如果两个类有彼此的引用，那就是双向关联，否则是单项关联。

数据库中的关联：

  1. 主键关联（两个表中的主键是一致的或者有关联关系）；
  2. （单项的）外键关联；
  3. （不常用）关联表（第三个表记录两个表之间的对应关系），如果是一对一可以这样的设计；

在类中，另一个类加的注释是OneToOne和JoinColumn，hibernate生成表的时候是进行的外键关联。

一对一双向外键关联

在OneToOne中使用mappedBy属性，声明是两边都有，但以对方为主，可以只生成一个外键。

只要有一对一双向外键关联就要写mappedBy属性。

注解是加在get方法之上的。

单双向在数据库中没有区别，有区别的是在java类中使用。

联合主键

此时的注解要使用JoinColumns组合。（不重要，但是要有点印象）

多对一单向关联

大多数情况的设计是在“多”的一方设计外键关联

设计范式

  1. 要有主键，列不可分
  2. 联合主键不能存在部分依赖
  3. 不能存在传递依赖

一对多的单项关联

在一的一方存在多的集合。一对多默认是多对多的一中特殊情况，需要OneToMany和JoinColumn的标签。

和数据库模型最匹配的集合：set，没有重复。

一对多多对一双向关联

首先在两方互相加入对方的引用，然后是用OneToMany，加入mappedby属性

单向关联

例子：老师和学生。加一张中间表即可。

面向对象中的单项关联，老师知道学生，但是学生不知道自己的所有老师。

使用JoinTable标签标签内部还有JoinColumn标签和InverseJoinColumns，能够改变中间表的列名。

双向关联

一般来说，多对多就比较少用，双向关联更加少用。

关联关系的CRUD

没有设置cascade不会进行关联关系的存储，只对cud有影响。fetch属性对于r有关系。

如果对象和对象之间的关系是双向的，一定要在程序中设置好关联。

多对多的双向关联，可能会存在懒加载的问题，当对象用到的时候才会加载到内存，但是这个时候，可能会出现另外一个问题：session已经断开了。

关联关系在删除的时候，需要先接触关联关系，否则会将有关联关系的数据一并删除。

集合映射

一对多的关系中，多的引用可以用set也可以用list，如果需要排序，那么用list。

对于map需要主要mapkey属性，但是用的也比较少。

继承映射

数据表设计思路：

  1. SingleTable，一张表中有所有子类的所有字段。
  2. TablePerClass，每个类一张表，但是这样的话没，代码中的多态会产生问题，而且多动态会有主见重复的问题。
  3. Joined，父类存储子类所有共有的属性，每个子类表存储自己特有的属性。

实际应用较少。

Hibernate查询语言

hibernate的查询方式，功能从强到弱依次为：

  1. NativeSQL，有点就是语言基础，实现功能多，确定就是不跨平台。
  2. HQL，hibernate自带的。
  3. EJBQL（JPQL），可以认为是HQL的子集。
  4. OBC
  5. OBE

前三种是大多数情况下能搞定。

新建Query的时候，引入包要注意引入Hibernate的包。

查询语句举例：

  1. Query q = session.createQuery(“from table”);
  2. Query q = session.createQuery(“from table t where t.name > ‘c’”);
  3. Query q = session.createQuery(“from table t order by c.name desc”);
  4. Query q = session.createQuery(“select distinct t from table t order by t.name desc”);
  5. Query q = session.createQuery(“from table t where t.id > :min and c.id < :max”);q.setParameter(“min”,2);q.setParameter(“max”,9);//也可以写成链式编程。
  6. Query q = session.createQuery(“from Topic t where t.category.id = 1”);//面向对象的，导航式的查询语言。
  7. Query q = session.createQuery(“select new ClassName(m.id,m.ame) from Msg”);
  8. Query q = session.createQuery(“select t.title, c.name from table t join t.category c”);
  9. Query q = session.createQuery(“select current_date, current_tim, current_timestamp from table”);//集群操作中，多个服务器围绕一个数据库进行服务，使用数据库的时间，服务器即可没有时间误差。

注意：in,exists如果需要二选一的话，用exists，因为效率高。

NamedQueries，可以之间通过名字进行集中管理，这个不是很重要。

性能优化

  1. 注意session.clear()的运用，尤其在不断分页循环的时候。应用：（1）在一个大集合中进行遍历，遍历msg，去除其中的含有敏感字样的对象。

java在代码层面上是没有的，但是java可能会引起内存泄露。语法级别上不用回收内存，比如连接池，文件，这些资源不关闭，一直存在。

1+N问题（面试常问）

取一个topic对象，在manytoone中设置了fatchtype=eager，导致和topic相关的对象都被取出来了。

解决方案：

  1. 在manytoone的注解中设置属性fetch = FetchType.LASY。
  2. 设置BatchSize(每次取多条)，但是解决的效果是1+(N/batchsize)
  3. 在查询语句中使用join fetch进行表连接。

以上解决方案中选1还是选3需要看是否需要关联对象的属性，如果需要使用3，不需要使用1。

list vs iterate

list将对象的全部信息取出来，iterate只取id，需要对象信息再通过id找到相关对象。在项目中iterate用的并不是很多。iterate会利用缓存。

原因：每次查询条件不一样，list是取出全部内容，很难重用，但是iterate发生机制可以利用缓存。

缓存（主要是在面试中有）

缓存：存储在内存中的本应存在硬盘中的内容。
使用：在hibernate.cfg.xml文件中添加相应的配置。
load方法默认使用二级缓存，iterate默认使用二级缓存。

开启查询缓存：1. 在配置文件中进行配置设置；2. 使用list或者session验证，可以确定是否已经开启了二级缓存。

缓存算法：

  1. LRU：Least Recently Used;
  2. LFU: Least Frequently Used（命中率高低）;
  3. FIFO: First In First Out;

事务的并发处理

事务：ACID（原子性，一致性，独立性，持久性），要么都完成，要么都不完成。

事务的并发处理相关问题：

  1. 丢失更新（支持事务的数据库不会出现）；
  2. 脏读，相当于读到了还没有提交的事务的数据；
  3. 不可重复读；
  4. 第二类丢失更新；
  5. 幻读。

数据库事务的隔离机制,一般情况下，设定为read-committed级别。避免了脏读，并发性较好。

乐观锁\悲观锁

悲观锁：在更新之前不会释放锁。
乐观锁：在程序中实现，增加一个version字段，如果有人更新，两个事务在更新之前是否有更新，如果有则修改了。

乐观锁不加锁，效率高。悲观锁加锁期间，有可能有事务更新，有可能没有事务更新。
