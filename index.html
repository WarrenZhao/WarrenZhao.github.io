博客显示图片

起因

博客好久不更新了，有一天更新了一篇带有数学公式的博客，为了让公式正常显示，结果……高挂了我的博客，正职找工作期间，感觉面试官访问不了我的博客好像我在说谎一样。于是，趁着今天有时间，我赶紧把博客恢复了，这一恢复不要紧啊，迁出了更多的问题，尤其是图片不能显示的问题……




简直就是拔出萝卜带出泥，一个问题还没有解决，又增加了新的问题……





本着一个码农应该有的钻研精神，我开始了漫长的搜索和尝试……


首先就是博客高挂了，github没办法显示，整了半天，又是删文件，又是改版本，就是不显示……




于是，我祭出大招——重装！


我的博客终于重见天日！

正题

接下来我就开始来解决图片不显示的问题。首先我看了看不显示图片的url，感觉应该是配置文件有问题，修改了url和root后无果，放弃此方法。在这里要吐槽一下官方文档，这个说明真的是太简单了！


然后再次翻阅官方文档和翻阅无数网页，最终我的解决方案是：

  1. 修改根目录下_config.yml文件中的post_asset_folder: false为post_asset_folder: true；
  2. 修改根目录下package.json文件中的hexo-asset-image": "^1.0.0为hexo-asset-image": "^0.0.1；
  3. 在根目录下执行npm install hexo-asset-image --save；
  4. 编辑带有图片的博客hexo clean && hexo g -d && hexo s尝试查看效果；

Mysql引擎

命令： show engines;

Mysql中支持的引擎有：InnoDB, MRG_MYISAM, MEMORY, BLACKHOLE, MyISAM, CSV, ARCHIVE, PERFORMANCE_SCHEMA, FEDERATED共九种，如下图：


根据上图说明，我们可知：
  1. InnoDB: 支持事务，支持行级锁，支持并发。
  2. MRG_MYISAM: 也称作merge引擎，是相同的MyISAM表的集合。
  3. MEMORY: 基于哈希的，存储在内存中，用于临时表。
  4. BLACKHOLE: 存储引擎，存什么都会消失。
  5. MyISAM: MyISAM存储引擎。
  6. CSV: CSV存储引擎。
  7. ARCHIVE: Archive存储引擎
  8. performance_SCHEMA: 用于监视Mysql服务器执行效果，收集Mysql服务器性能参数的数据库。
  9. FEDERATED: 存储引擎访问在远程数据库的表中的数据，而不是本地的表。

使用场景：

引擎 特点 使用场景
InnoDB 事务性，支持ACID；
redo log实现事务的持久性，undo log实现事务的原子性，存储未完成的事务log；
支持行级锁，最大程度上支持了并发。		 要求事务、可靠性高、更新和查询频繁且锁行的机会比较大的场景
MyISAM 由.myd数据文件和.myi索引文件组成；
表级别锁，写入与读互斥；
支持全文索引，支持表压缩，只读不可以写入。		 没有事务、只读类的应用、空间类型应用，做很多Count计算的
CSV 以CSV格式进行存储，以文本方式存储在文件中；
.csv存内容，.csm存储元数据，.frm存储表结构；
列不为NULL，不支持索引		 数据交换的中间表
Archive 以zlib对标数据进行压缩，磁盘IO更好；
表文件格式.arz, a.frm;
只支持insert和select操作，允许在自增id列上加索引。		 日志和数据采集类应用
Memory heap存储引擎，数据保存在内存中，重启后数据丢失；
支持hash索引和BTree索引；
使用表级锁；
不支持blob和text等大字段，默认表大小16M		 查找映射表，保存中间结果表和缓存结果表
FEDERATED 本地之存储表结构，查询时通过建表时使用的链接符去获取远程库的数据返回到本地. 应用需要使用多库，但是不想设计为链接多库，设计使用为表映射.
